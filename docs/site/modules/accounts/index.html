💳 ACCOUNT - Módulo de Cuentas

Guía Completa de Desarrollo de Historias de Usuario

📌 Propósito: Gestión completa del ciclo de vida de cuentas de tarjetas de crédito, incluyendo visualización de información financiera y actualización transaccional de datos de cuenta y cliente

📋 Resumen Ejecutivo del Módulo

🎯 Contexto de Negocio: El módulo ACCOUNT es el núcleo operativo para la gestión de cuentas de tarjetas de crédito. Permite a representantes de servicio al cliente y administradores realizar consultas detalladas, visualizar estados financieros y actualizar información crítica de cuentas y clientes en tiempo real.

🔑 Responsabilidades Principales

  • ✅ Búsqueda Avanzada: Localización de cuentas por ID de 11 dígitos con validación instantánea
  • ✅ Visualización Financiera: Despliegue completo de límites de crédito, balances actuales y ciclos de facturación
  • ✅ Actualización Transaccional: Modificación atómica de datos de cuenta y cliente con garantía ACID
  • ✅ Seguridad PCI-DSS: Protección y enmascarado de datos sensibles (SSN, números de tarjeta)
  • ✅ Validación en Tiempo Real: Verificación instantánea de reglas de negocio (FICO, ZIP codes, estados)
  • ✅ Gestión de Estados: Control del ciclo de vida de cuentas (activo/inactivo)

⚡ Factores de Aceleración de Desarrollo

🚀 Reutilización de Componentes: El módulo ACCOUNT proporciona múltiples componentes y patrones reutilizables que aceleran significativamente el desarrollo de nuevas funcionalidades.

  • 🎣 useAccountView Hook (Custom): Hook reutilizable que encapsula toda la lógica de búsqueda, manejo de estados (loading/error/data) y validación inline. Ahorra ~40 líneas de código por implementación. Reduce desarrollo en 60%.
  • 🎣 useAccountUpdate Hook (Custom): Detecta cambios automáticamente comparando estado original vs actual mediante serialización JSON. Elimina necesidad de tracking manual de cada campo. Reduce complejidad de formularios de edición en 50%.
  • 🛡️ AccountValidationService (Backend): Validaciones centralizadas en un solo servicio. Nuevas validaciones se agregan una sola vez y se aplican en todos los endpoints. Evita duplicación de lógica de validación.
  • 🎨 Material-UI Components: TextField, Card, Button, Dialog, Chip preconstruidos con estilos consistentes. Reduce tiempo de desarrollo de UI en 60-70%. No requiere CSS personalizado.
  • 🔌 API RESTful Documentada: 4 endpoints bien definidos (GET /account-view, GET /account-view/initialize, GET /accounts/{id}, PUT /accounts/{id}) facilitan integración frontend-backend en 1-2 días.
  • 🎭 MSW (Mock Service Worker): Mocks completos permiten desarrollo frontend sin backend activo. 10 cuentas de prueba precargadas. Acelera testing y desarrollo paralelo.
  • 📐 TypeScript Types: Interfaces AccountViewResponse y AccountUpdateData completamente tipadas. Previene errores en tiempo de compilación. IntelliSense completo en IDEs.

📋 Reglas de Negocio

Identificación y Búsqueda

RN-001: El Account ID debe ser exactamente de 11 dígitos numéricos
RN-002: El Account ID no puede ser todo ceros (00000000000)
RN-003: La búsqueda valida existencia en: CardXrefRecord → Account → Customer

Seguridad y Privacidad

RN-006: El SSN se muestra enmascarado: ***-**-XXXX (solo últimos 4 dígitos)
RN-007: El número de tarjeta se enmascara: ****-****-****-XXXX

Validaciones Críticas

RN-009: Active Status solo acepta 'Y' (activo) o 'N' (inactivo)
RN-012: FICO Score debe estar en rango 300-850
RN-015: ZIP Code formato: ^\d{5}(-\d{4})?$

Transaccionalidad

RN-018: Actualización de Account y Customer es atómica (todo o nada)
RN-021: Se hace "READ FOR UPDATE" (lock pesimista) antes de actualizar

🚨 Riesgos de Desarrollo y Mitigaciones

⚠️ Performance en Búsquedas

Riesgo: Búsqueda secuencial en 3 tablas puede degradarse con alto volumen

Mitigación: Implementar índices en accountId y customerId; considerar caché Redis para cuentas frecuentes

⚠️ Falta de i18n

Riesgo: Mensajes hardcodeados en inglés no cumplen requisito de español

Mitigación: Implementar react-i18next antes de nuevas funcionalidades (estimado 5 puntos)

⚠️ Sin Auditoría de Cambios

Riesgo: No hay trazabilidad de quién modificó qué valores

Mitigación: Implementar Audit Trail con Spring Data Envers (5 puntos)

⚠️ Validaciones COBOL Comentadas

Riesgo: Validación de SSN está comentada en código migrado

Mitigación: Revisar y habilitar o reemplazar con validaciones modernas

// useAccountView.ts - Patrón reutilizable
export function useAccountView() {
  const [data, setData] = useState<AccountViewResponse | null>(null);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState<string | null>(null);

  const searchAccount = async (request: AccountViewRequest) => {
    setLoading(true);
    try {
      const accountIdNum = parseInt(request.accountId);
      if (isNaN(accountIdNum) || accountIdNum === 0) {
        throw new Error("Account must be non-zero 11 digit number");
      }
      
      const response = await api.get<AccountViewResponse>(
        `/account-view?accountId=${request.accountId.padStart(11, "0")}`
      );
      setData(response.data);
    } catch (err) {
      setError(err.message);
    } finally {
      setLoading(false);
    }
  };

  return { data, loading, error, searchAccount };
}

Patrón Backend: Servicio Transaccional

// AccountUpdateService.java
@Service
public class AccountUpdateService {
  
  @Transactional  // ← Clave: garantiza atomicidad
  public Map<String, String> updateAccountAndCustomer(
      Long accountId, AccountUpdateRequestDto request) {
    
    // Leer con lock
    Account account = accountRepository.findById(accountId)
        .orElseThrow(() -> new ResourceNotFoundException("Account not found"));
    Customer customer = customerRepository.findByAccountId(accountId)
        .orElseThrow(() -> new ResourceNotFoundException("Customer not found"));
    
    // Validar
    List<String> errors = validationService.validateAccountUpdate(request);
    if (!errors.isEmpty()) {
      throw new ValidationException(String.join(", ", errors));
    }
    
    // Actualizar
    updateAccountFromRequest(account, request);
    updateCustomerFromRequest(customer, request);
    
    // Guardar (si falla uno, rollback automático)
    accountRepository.save(account);
    customerRepository.save(customer);
    
    return Map.of("message", "Account updated successfully");
  }
}
