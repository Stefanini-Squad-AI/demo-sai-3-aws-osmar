💳 CREDIT CARD - Tarjetas

Guía de Desarrollo de User Stories

Módulo completo para gestión del ciclo de vida de tarjetas de crédito

📋 Descripción General

ID del Módulo: credit-card

Propósito: Gestión completa del ciclo de vida de tarjetas de crédito vinculadas a cuentas de clientes.

Contexto de Negocio: Permite a representantes de servicio y administradores listar, visualizar y actualizar información de tarjetas de crédito, incluyendo estado, nombre grabado, y fecha de expiración. El módulo implementa control de acceso por roles y validaciones de negocio estrictas.

⚡ Características Destacadas:
  • Paginación inteligente (7 registros por página)
  • Control de acceso por tipo de usuario (Admin/User)
  • Máquina de estados para flujo de actualización
  • Validaciones en tiempo real con feedback inmediato
  • Mapeo seguro entre estados del backend (CardStatus enum) y UI

✅ Criterios de Aceptación y Patrones

🎯 Criterios de Aceptación Tipo

Para Listado de Tarjetas:

  • DADO que soy un usuario no-admin CUANDO busco tarjetas sin Account ID ENTONCES debe mostrar error "Account ID is required"
  • DADO que hay más de 7 resultados CUANDO cargo la página 1 ENTONCES debe mostrar los primeros 7 y habilitar botón "Next"
  • DADO que estoy en la última página CUANDO presiono "Next" ENTONCES el botón debe estar deshabilitado

Para Visualización de Detalles:

  • DADO que ingreso Account ID válido y Card Number válido CUANDO presiono "Search" ENTONCES debe mostrar tarjeta con CVV Code, nombre grabado y fecha de expiración
  • DADO que el Account ID tiene menos de 11 dígitos CUANDO presiono "Search" ENTONCES debe mostrar error "Account number must be a non zero 11 digit number"

Para Actualización de Tarjetas:

  • DADO que cambio el nombre grabado CUANDO presiono "Validate" ENTONCES el nombre debe convertirse a mayúsculas y validar solo letras/espacios
  • DADO que cambio la fecha de expiración a una fecha pasada CUANDO presiono "Validate" ENTONCES debe mostrar error "Card expiry date cannot be in the past"
  • DADO que confirmo cambios válidos CUANDO presiono "Confirm" ENTONCES debe actualizar en backend y mostrar mensaje de éxito

🚨 Consideraciones y Riesgos

⚠️ Riesgos Técnicos Identificados

1. Seguridad del CVV

Riesgo: CVV Code expuesto en respuestas de API sin control de acceso

Impacto: Alto - Vulnerabilidad de seguridad PCI-DSS

Mitigación: Implementar enmascaramiento en backend, mostrar solo con autenticación adicional (8 puntos)

2. Falta de Internacionalización (i18n)

Riesgo: Todos los mensajes y labels están hardcodeados en inglés

Impacto: Medio - Dificulta adopción en mercados no angloparlantes

Mitigación: Implementar react-i18next con archivos de traducción (5 puntos)

3. Sin Auditoría de Cambios

Riesgo: No hay trazabilidad de quién modificó qué y cuándo

Impacto: Alto - Incumplimiento de compliance y auditoría

Mitigación: Implementar Audit Trail con Spring Data Envers (5 puntos)

4. Paginación en Memoria

Riesgo: Rendimiento degradado con miles de tarjetas por cuenta

Impacto: Medio - Experiencia de usuario deficiente en cuentas grandes

Mitigación: Implementar cursor-based pagination en backend (3 puntos)

💡 Tech Debt Identificado

  • Mapeo de Estados: Actualmente manual entre backend enum y UI - Ya mitigado con funciones helper
  • Validaciones Duplicadas: Algunas validaciones están repetidas entre hooks - Pendiente refactorización (2 puntos)
  • MSW Handlers: Estado compartido entre handlers puede causar inconsistencias en tests paralelos (1 punto)
