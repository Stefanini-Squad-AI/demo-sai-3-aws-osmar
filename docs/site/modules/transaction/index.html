💸 TRANSACTION

Transacciones financieras – Guía práctica para escribir User Stories.

Propósito: Registrar, auditar, consultar y reportar movimientos de tarjetas dentro de CardDemo con vínculos claros a servicios backend.

📋 Resumen rápido

Responsabilidades clave: listado paginado (F7/F8), vista por Transaction ID (F5 vuelve al listado), registro manual con confirmación (Y), reportes mensuales/anuales/personalizados, exportación TXT e información de merchant al detalle.

⚙️ Stack operativo

Frontend: React 18 + TypeScript + Material-UI 5

Estado: Redux + Hooks

Backend: Spring Boot (endpoints `/transactions*`, `/transaction-view`, `/v1/reports/transactions/*`).

📐 Métricas de impacto

  • 90% de agentes usan este módulo
  • 40 búsquedas/usuario/día
  • -60% errores de captura manual

🔗 Factores de acceso

Access control con Auth Module y validaciones contextuales de cuentas y tarjetas.

MSW mantiene 50 transacciones mock para asegurar coherencia local.

🔧 Fundamento técnico

TransactionListScreen

Renderiza tabla, chips de página, validaciones numéricas y botones F7/F8/Enter.

Se apoya en `useTransactionList` para mutaciones POST.

TransactionViewScreen

Formulario read-only que formatea montos con `Intl.NumberFormat` y muestra metadata de `TransactionViewResponse`.

TransactionAddScreen

Formulario dividido, datos de prueba, validaciones manuales y confirmación `Y` antes de invocar el backend.

TransactionReportsScreen

Genera reportes mensuales/anuales/personalizados, muestra dialogo con tabla sticky y descarga TXT.

🌐 Interfaces públicas

POST /transactions/list            - Lista transacciones (body: transactionId?, pageNumber)
POST /transactions/next-page      - Página siguiente (keyset pagination con lastTransactionId)
POST /transactions/previous-page  - Página anterior (keyset con firstTransactionId)
GET  /transaction-view/search?transactionId={id} - Detalle + metadata (currentDate, programName)
POST /transaction/clear            - Inicializa la vista (CT01)
POST /transactions                - Agrega movimiento (confirmación Y)
POST /v1/reports/transactions/{monthly|yearly|custom} - Reportes con confirmación

📋 Reglas de negocio

RN-TR-001: Transaction ID numérico obligatorio para vista y listado.
RN-TR-002: Registro manual solo con `confirmation = 'Y'`.
RN-TR-003: Monto en rango [-99,999,999.99, +99,999,999.99] y fechas en `YYYY-MM-DD`.
RN-TR-004: Requiere Account ID (11 dígitos) o Card Number (16 dígitos).

✅ Criterios de aceptación

Validación

Dado que digito un Transaction ID, cuando es inválido, entonces muestro error en el campo y no inicio la petición.

Registro

El formulario borra los campos y muestra mensaje de éxito cuando `POST /transactions` responde `success = true`.

Reportes

Al confirmar un reporte mensual, debe aparecer el dialogo con la tabla y permitir descargar TXT/print.

Performance

La respuesta de búsqueda debe ser menor a 400ms (P95) y mostrar loader mientras se espera.

⚡ Presupuestos de rendimiento

  • Listados: < 600ms (P95) para páginas de 100 registros.
  • Búsqueda por ID: < 400ms (P95).
  • Registro manual: < 1s con validaciones incluidas.
  • Reportes: < 2s por reporte mensual (carga + render dialog + TXT).

🚧 Riesgos y mitigaciones

Dependencia de keyset pagination

Riesgo de loops si el backend no envía `firstTransactionId`/`lastTransactionId`.

Mitigación: Validar su existencia antes de navegar y mostrar mensaje.

Validaciones divergentes

Regex de monto o fechas puede hacer que el frontend acepte datos que el backend rechaza.

Mitigación: Replicar exactamente las expresiones de validación del backend y documentarlas en `useTransactionAdd`.

Reportes pesados

Riesgo de cargar demasiados datos sin límites en `custom`.

Mitigación: Restringir rangos de fecha (ej. 30 días) y mostrar spinner mientras se genera el reporte.
